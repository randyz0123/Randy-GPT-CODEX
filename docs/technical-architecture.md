# xiaofen.ai 技术架构说明

## 推荐技术架构
采用“前后端分离 + Agent 编排 + RAG 知识检索”的模块化架构：

1. **应用层**：销售工作台、管理后台
2. **服务层**：用户与组织、线索与跟进、分析看板、知识库管理
3. **智能层**：对话建议 Agent、质检与复盘 Agent
4. **数据层**：业务库（PostgreSQL）+ 缓存（Redis）+ 向量检索（pgvector）
5. **集成层**：企业微信/CRM/埋点分析接口

该架构优点：
- MVP 可以快速落地
- 后续可按模块独立扩容
- 对 AI 能力与业务能力进行边界隔离，便于稳定迭代

## 前端架构
- **框架**：Next.js（App Router）+ TypeScript
- **UI**：Tailwind CSS + shadcn/ui
- **状态管理**：React Query + Zustand（必要场景）
- **权限控制**：基于组织角色的路由守卫与组件级权限渲染
- **页面结构**：
  - 销售工作台：线索列表、对话建议、跟进任务
  - 管理后台：转化看板、团队排行、知识库配置

## 后端架构
- **服务框架**：NestJS（模块化）
- **接口协议**：REST API（MVP），后续可补充 GraphQL
- **核心模块**：
  - Auth 模块：登录鉴权、组织与角色
  - Lead 模块：线索与阶段管理
  - Conversation 模块：沟通记录与建议生成
  - Workflow 模块：跟进任务与自动提醒
  - Analytics 模块：核心经营指标计算
  - Knowledge 模块：文档入库与检索
  - Agent Orchestrator：Prompt 编排与策略路由

## 数据库模型
建议核心实体如下：

- `organizations`：企业主体
- `users`：用户信息
- `roles`：角色定义（管理员/主管/销售）
- `leads`：线索信息（来源、意向度、阶段）
- `conversations`：会话记录（时间、文本、渠道）
- `follow_up_tasks`：跟进任务（状态、负责人、截止时间）
- `knowledge_docs`：知识文档元数据
- `knowledge_chunks`：切片内容与向量索引
- `agent_logs`：Agent 建议日志（输入、输出、采纳情况）
- `metrics_daily`：日报级指标快照

建模原则：
- 先满足可追溯与可分析，再追求复杂自动化
- 对 AI 输出做日志留存，便于审计与优化

## AI Agent 模块设计

### 1) 对话建议 Agent
输入：客户上下文、会话历史、当前阶段、知识检索结果。
输出：候选回复、建议意图、风险提示。

### 2) 异议处理 Agent
聚焦高频异议场景（价格、效果、时机、信任），输出结构化应答方案。

### 3) 复盘诊断 Agent
按团队/个人周期输出“有效动作、失分点、优化建议”。

### Agent 设计原则
- “建议优先”而非“强制自动发送”
- 可解释（建议来源可追踪）
- 可配置（按行业与团队策略微调）

## 知识库模块设计

### 数据来源
- 历史聊天记录
- 标准话术文档
- 产品手册与常见问题

### 处理流程
1. 清洗脱敏
2. 语义切片
3. 向量化入库
4. 检索召回
5. 重排与注入 Prompt

### 质量策略
- 设定文档有效期与责任人
- 支持版本化，避免旧话术污染建议
- 记录“建议采纳率”反向评估知识质量

## 权限与安全设计

### 权限模型
- 多租户隔离（Organization 级别）
- RBAC 角色权限（管理员/主管/销售）
- 数据范围控制（仅可见本组织或授权团队数据）

### 安全策略
- 全链路 HTTPS
- 关键字段加密存储（手机号、客户识别信息）
- 操作日志与审计追踪
- Prompt 注入与越权访问防护
- 定期备份与灾难恢复演练

### 合规建议
- 明确告知 AI 建议为辅助，不替代人工决策
- 对敏感信息处理设置最小化原则
- 与客户签署数据处理协议（DPA）
